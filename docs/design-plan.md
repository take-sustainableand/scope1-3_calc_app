# S&Carbon Scope 1-3 算定ツール 設計ドキュメント / 実装Plan

- **作成日**: 2026-04-26
- **対象**: GitHub Pages 上で個人利用する Scope 1・2・3 排出量算定アプリ
- **前提**: 原単位は利用者が必要なものを手動登録する。外部データ連携は行わない。

## 1. 目的

スコープ1、2、3の活動量データと手動登録した原単位から CO2e 排出量を算定し、ダッシュボード、入力、原単位管理、分析、レポート、目標・施策・アラート管理をスマホとパソコンの両方で確認できる UI を提供する。

GitHub Pages で静的配信し、個人利用に必要なデータは GitHub リポジトリ内の JSON として保存・参照できるライトな構成を目指す。

## 2. 画面設計

画像から確認できた画面:

- ダッシュボード
- データ入力
- 原単位管理
- 排出量分析
- 設定

不足分として判断して追加した画面:

- データ一覧: 登録済み活動データの確認、JSON出力
- レポート: 月次サマリー、Scope別明細、原単位マスタの出力導線
- 目標・進捗管理: 2030年削減目標、進捗、必要削減量
- 削減施策管理: 検討中、実行中、完了の簡易カンバン
- アラート管理: 未入力、変動、原単位未設定の確認

PC版は左サイドバー、上部フィルター、カード型ダッシュボードを基本にする。スマホ版は上部ブランドヘッダー、下タブ、縦積みカードを基本にする。分析画面は画像に合わせてダークテーマの分析UIを採用する。

## 3. データモデル

### EmissionFactor

```json
{
  "id": "f-electricity",
  "scope": "Scope 2",
  "name": "電力（日本）",
  "category": "電力",
  "unit": "kWh",
  "coefficient": 0.000456,
  "source": "手動登録",
  "region": "日本",
  "year": "2023",
  "status": "公式"
}
```

### ActivityRecord

```json
{
  "id": "a-001",
  "factorId": "f-electricity",
  "amount": 7039474,
  "site": "全拠点",
  "supplier": "電力会社",
  "date": "2026-04-30",
  "memo": "電力使用量"
}
```

### AppSettings

```json
{
  "theme": "light",
  "site": "すべてのサイト",
  "period": "2026年4月",
  "githubOwner": "",
  "githubRepo": "",
  "dataPath": "data/scarbon-state.json"
}
```

算定式:

```text
排出量（t-CO2e） = 活動量 × 原単位係数
```

## 4. GitHub Pages / GitHub保存設計

推奨構成:

```text
/
  index.html
  styles.css
  app.js
  docs/design-plan.md
  data/
    scarbon-state.json
```

読み取り:

- GitHub Pages 配信時は `data/scarbon-state.json` を `fetch()` して初期データとして読み込む。
- ブラウザ内編集内容はまず `localStorage` に保存する。
- JSON出力により手動バックアップできるようにする。

書き込み:

- GitHub Contents API を利用して `data/scarbon-state.json` を更新する。
- Personal Access Token はコードやリポジトリに保存しない。
- 利用者がブラウザの設定画面で入力し、ブラウザ内ストレージにのみ保持する。
- 書き込み時は `sha` を取得し、JSON全体を更新する1ファイル方式にする。

個人利用のため、初期実装では次の順で軽量に進める。

- Phase 1: `localStorage` + JSON出力
- Phase 2: `data/scarbon-state.json` の読み込み
- Phase 3: GitHub Contents API による保存
- Phase 4: 競合検知、履歴表示、復元

## 5. セキュリティ要件

- APIキー、トークン、認証情報をリポジトリへコミットしない。
- GitHub token は Fine-grained token を使い、対象リポジトリと Contents read/write のみに限定する。
- 公開リポジトリに保存する場合、請求書番号、個人名、取引先の機密情報などは保存しない。
- 証憑ファイル本体は初期スコープ外とし、必要ならファイル名や参照メモだけ保存する。
- 外部原単位DBや外部APIとの連携は行わない。

## 6. 実装Plan

1. 静的UI基盤を作る
   - `index.html`, `styles.css`, `app.js` を用意する。
   - 画面イメージに合わせてPC版とスマホ版をレスポンシブ実装する。
   - 成功定義: 主要10画面へ遷移でき、スマホ幅とPC幅で破綻なく表示される。

2. 手動登録データのローカル保存を実装する
   - 原単位登録フォームを実装する。
   - 活動量入力フォームを実装する。
   - `localStorage` に原単位と活動データを保存する。
   - 成功定義: リロード後も登録した原単位と活動データが残る。

3. 算定ロジックを分離する
   - `amount × coefficient` の基本算定を共通関数化する。
   - Scope別、カテゴリ別、拠点別の集計関数を用意する。
   - 成功定義: 同一データからダッシュボード、一覧、分析の値が一致する。

4. GitHub上のJSON参照を実装する
   - `data/scarbon-state.json` を作成する。
   - 初回ロード時にJSONを読み込み、失敗時はローカルデータへフォールバックする。
   - 成功定義: GitHub Pages配信後もJSONデータが画面に反映される。

5. GitHub保存を実装する
   - 設定画面に owner / repo / branch / path / token 入力を追加する。
   - GitHub Contents API で `GET` して `sha` を取得する。
   - 更新JSONを base64 化して `PUT` する。
   - 成功定義: アプリから保存した内容が GitHub リポジトリのJSONにコミットされる。

6. レポート出力を実装する
   - JSON出力を安定化する。
   - CSV出力を追加する。
   - 月次・Scope別・原単位マスタの出力テンプレートを作る。
   - 成功定義: 登録データを別環境で再取り込みできる。

7. 品質改善と公開準備
   - アクセシビリティ、キーボード操作、空状態、入力エラーを調整する。
   - GitHub Pages の公開設定手順をREADME化する。
   - 成功定義: 個人利用に必要な登録、確認、保存、復元が一通り完結する。

## 7. 検証項目

### UI / レスポンシブ

- 375px幅で下タブが表示され、カードが1列で崩れない。
- 768px幅でカードが読みやすく折り返される。
- 1280px以上で左サイドバー、上部フィルター、複数列カードが表示される。
- ボタン内テキストがはみ出さない。
- 分析画面のダークテーマで文字コントラストが十分にある。

### 入力 / 算定

- Scope切替で該当Scopeの原単位だけ選択できる。
- 活動量を変更すると推定排出量が更新される。
- 原単位を新規登録し、活動入力で利用できる。
- 0以下や未入力の活動量は保存できない。
- Scope別合計と総排出量が正しく再計算される。

### 保存 / 復元

- `localStorage` 保存後、リロードしてもデータが残る。
- JSON出力が有効なJSONとして開ける。
- GitHub JSON読み込み失敗時にアプリが停止しない。
- GitHub保存時、古い `sha` の競合を検出できる。
- token がコード、ログ、出力ファイルに混入しない。

### GitHub Pages

- ルート直下の `index.html` から表示できる。
- 外部CDNなしで表示できる。
- GitHub Pages のサブパス配信でも CSS/JS が読み込まれる。
- 公開リポジトリで機微情報が露出しない。

### 運用

- 月次データ入力からダッシュボード確認まで迷わず進める。
- 原単位を手動登録するだけで新しい排出源に対応できる。
- レポートに必要なScope別、カテゴリ別、拠点別集計を確認できる。
- 削減目標と施策が排出量確認画面から辿れる。
